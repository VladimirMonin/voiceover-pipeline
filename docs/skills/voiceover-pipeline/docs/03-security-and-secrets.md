# Безопасность и секреты: .env и API-ключи

> АГЕНТ: ЧИТАЙ ЭТОТ ФАЙЛ ЦЕЛИКОМ И СОБЛЮДАЙ ПРАВИЛА.
> Это критичный файл. Нарушение любого правила — ошибка навыка.

## Главное правило

**НИКОГДА не читай содержимое `.env`.**
Даже чтобы проверить, есть ли там ключ.
Даже чтобы убедиться, что пользователь уже вписал значение.
Даже если пользователь просит «посмотри, правильно ли я вставил ключ».

Вместо этого используй `voiceover doctor --json` — он проверит наличие ключей
без раскрытия их значений.

## Второе правило

**НИКОГДА не проси пользователя прислать API-ключ в чат.**
Ключ вставляется только в `.env` файл, локально.
Если пользователь прислал ключ в чат — не читай, попроси вставить в `.env`.

## Пошаговый протокол для агента

### Шаг 1: Убедись что `.env` в `.gitignore`

Проверь `.gitignore` в корне проекта. Если файла `.gitignore` нет — создай его
(даже если это не git-репозиторий, это безопасная защита на будущее).
Если в `.gitignore` нет строки `.env` — добавь.

### Шаг 2: Создай `.env.example`

Создай файл `.env.example` с безопасными mock placeholder-ами (не с реальными ключами):

```env
# Copy this file to .env and fill real keys.
# .env is searched in CWD and upwards through parent directories.
# Never commit .env.

OPENROUTER_API_KEY=sk-or-v1-xxxxxxxxxxxxxxxx
POLZA_API_KEY=pza_xxxxxxxxxxxxxxxx
```

Создай `.env` из `.env.example`: `Copy-Item .env.example .env` (Windows) или `cp .env.example .env` (macOS/Linux).
Сообщи пользователю: «Я создал `.env` из шаблона. Открой `.env` и впиши свои API-ключи.»

### Шаг 3: Объясни какие ключи нужны

В зависимости от выбранного провайдера:

- **Polza (chat-audio + tts):** `POLZA_API_KEY` — начинается с `pza_`. Один ключ для обоих polza-провайдеров.
- **OpenRouter:** `OPENROUTER_API_KEY` — начинается с `sk-or-v1-`
- **Qwen-local:** ключи НЕ нужны (работает локально на GPU)

### Шаг 4: Попроси один раз

Скажи пользователю: «Открой `.env` и впиши ключ. Больше я спрашивать не буду.»

После этого НЕ спрашивай ключи повторно. Если `doctor` показывает missing key,
просто напомни: «Проверь `.env`, ключ всё ещё не найден.»

### Шаг 5: Проверь через doctor

```powershell
voiceover doctor --provider polza-chat-audio --json
voiceover doctor --provider openrouter-tts --json
```

Смотри на поля `polza_key.ok` и `openrouter_key.ok` в JSON-ответе.
Если `ok: false` и `required: true` — скажи пользователю какой ключ добавить.
Если `required: false` — ключ не нужен для данного провайдера, это не ошибка.

### Шаг 6: Дальше работай молча

После того как ключи есть — не спрашивай о них снова.
Просто запускай команды. Если ключ исчез — CLI сам упадёт с exit code 20.

## Где ищется .env

`.env` ищется в текущей рабочей директории (CWD), затем в родительских
директориях вверх по дереву. Это позволяет держать один `.env` в корне проекта
и запускать CLI из любой поддиректории.

## Чего нельзя делать

- ❌ `cat .env` / `type .env` / чтение `.env` любым инструментом
- ❌ Показывать содержимое `.env` в ответе
- ❌ Просить пользователя выполнить `export POLZA_API_KEY=...` в терминале
- ❌ Коммитить `.env` в git
- ❌ Создавать `.env` с реальными ключами (только `.env.example` с placeholder-ами)
- ❌ Просить ключ повторно после того как `doctor` показал `ok: true`
- ❌ Угадывать или подбирать ключи
